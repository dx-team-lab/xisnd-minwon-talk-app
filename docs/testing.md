# 자기검증 체크리스트

> 이 문서는 각 개발 단계 완료 후, 다음 단계로 넘어가기 전에
> **스스로 점검**해야 할 체크리스트입니다.
>
> ⚠️ **중요 원칙**: 보안 항목과 DB 항목은 AI 자기검증만으로 PASS로 보지 않습니다.
> 이 항목들은 반드시 **개발자 검토**를 받은 후 PASS로 처리합니다.

---

## 단계별 체크리스트 개요

| 단계 | 작업 내용 | 검증 주체 |
|------|---------|---------|
| 1단계 | DB 스키마 설계 | AI 초안 → 개발자 최종 확인 |
| 2단계 | Spring Boot API 개발 | AI 자기검증 + 개발자 확인 |
| 3단계 | 보안 설정 (JWT, CORS, 권한) | 반드시 개발자 검토 |
| 4단계 | 프론트엔드 API 연동 | AI 자기검증 |
| 5단계 | 전체 통합 테스트 | AI 자기검증 + 개발자 최종 확인 |

---

## 1단계: DB 스키마 설계 완료 체크리스트

### 기본 구조
- [ ] Firebase 컬렉션 목록이 빠짐없이 MySQL 테이블로 매핑되었다.
- [ ] 모든 테이블에 `created_at`, `updated_at`, `created_by`, `updated_by` 공통 컬럼이 있다.
- [ ] 삭제가 필요한 테이블에 `deleted_yn`, `deleted_at` 논리 삭제 컬럼이 있다.
- [ ] 모든 테이블의 PK가 `BIGINT AUTO_INCREMENT`로 설정되어 있다.
- [ ] 모든 테이블에 `utf8mb4` 문자셋이 적용되어 있다.

### 정규화
- [ ] Firestore 배열 필드(`type`, `requestContent` 등)가 별도 테이블로 분리되었다.
- [ ] Firestore 서브컬렉션(`complaints`, `siteImages`)이 별도 테이블 + FK로 설계되었다.
- [ ] 코드성 데이터(지역, 유형, 단계 등)가 `code_detail` 테이블로 관리된다.

### ⚠️ 개발자 검토 필요
- [ ] 테이블 간 FK 제약조건 적용 여부가 결정되었다 (DB 레벨 vs 논리적 FK).
- [ ] 인덱스 설계가 실제 조회 패턴에 맞게 검토되었다.
- [ ] **개발자 검토 완료** ☐

## 2단계: Spring Boot API 개발 완료 체크리스트

### 코드 구조
- [ ] Controller → Service → Repository 레이어 구조를 따른다.
- [ ] Controller에서 Repository를 직접 호출하지 않는다.
- [ ] Entity를 API 응답으로 직접 반환하지 않고 DTO로 변환한다.
- [ ] 모든 쓰기 Service 메서드에 `@Transactional`이 적용되어 있다.
- [ ] 조회 전용 메서드에 `@Transactional(readOnly = true)`가 적용되어 있다.

### API 규칙
- [ ] 모든 API URL이 `/api/v1/` 로 시작한다.
- [ ] 공통 응답 포맷 (`success`, `code`, `message`, `data`)을 사용한다.
- [ ] 에러 응답도 공통 포맷으로 반환한다.
- [ ] Request DTO에 Validation 어노테이션(`@NotBlank`, `@Size` 등)이 적용되어 있다.

### 로그
- [ ] `System.out.println`이 사용되지 않는다.
- [ ] Slf4j 로그 레벨이 적절하게 사용된다 (INFO/WARN/ERROR).
- [ ] 로그에 비밀번호, 토큰, 개인정보가 출력되지 않는다.

---

## 3단계: 보안 설정 체크리스트

> ⚠️ **이 단계는 AI 자기검증만으로 PASS를 인정하지 않습니다.**
> 아래 항목은 반드시 개발자가 직접 확인해야 합니다.

### CORS
- [ ] `allowedOrigins("*")` 가 운영 설정에 없다.
- [ ] 허용 Origin이 환경 변수로 관리된다.
- [ ] OPTIONS 요청에 정상 응답한다.
- [ ] **개발자 검토 완료** ☐

### JWT 인증
- [ ] 로그인 없이 보호된 API 호출 시 `401` 응답이 반환된다.
- [ ] 만료된 토큰으로 요청 시 `401` 응답이 반환된다.
- [ ] JWT Secret Key가 코드에 하드코딩되지 않았다.
- [ ] **개발자 검토 완료** ☐

### API 접근 권한
- [ ] MANAGER 계정으로 ADMIN 전용 API 호출 시 `403` 응답이 반환된다.
- [ ] 미승인 사용자가 데이터 API에 접근 시 `403` 응답이 반환된다.
- [ ] **개발자 검토 완료** ☐

### 민감정보
- [ ] MANAGER 계정으로 조회 시 보상금액이 마스킹된다.
- [ ] MANAGER 계정으로 조회 시 품의 링크(details)가 마스킹된다.
- [ ] 마스킹 처리가 서버 응답에서 이루어진다 (클라이언트 처리 아님).
- [ ] **개발자 검토 완료** ☐

## 4단계: 프론트엔드 API 연동 체크리스트

### API 통신
- [ ] Firebase SDK 직접 호출 코드가 `src/services/` 의 API 서비스 함수로 교체되었다.
- [ ] 모든 API 요청에 `Authorization: Bearer {token}` 헤더가 포함된다.
- [ ] 401 응답 시 로그인 페이지로 리다이렉트한다.
- [ ] 403 응답 시 적절한 안내 메시지가 표시된다.

### UI 상태
- [ ] 데이터 로딩 중 스피너(로딩 인디케이터)가 표시된다.
- [ ] 에러 발생 시 사용자에게 안내 메시지가 표시된다.
- [ ] MANAGER 역할에서 보상금액 필드가 화면에 `[권한 필요]` 또는 마스킹으로 표시된다.
- [ ] MANAGER 역할에서 품의 링크가 화면에 표시되지 않는다.

### 코드 품질
- [ ] `any` 타입 사용 없이 API 응답 타입이 정의되어 있다.
- [ ] API 서비스 함수가 `src/services/` 폴더에 분리되어 있다.
- [ ] Firebase 관련 import가 남아있지 않다.

## 5단계: 전체 통합 테스트 체크리스트

### 핵심 기능 동작 확인
- [ ] 로그인 → 대시보드 진입이 정상 동작한다.
- [ ] 사례 등록 → 목록 조회 → 수정 → 삭제가 정상 동작한다.
- [ ] 엑셀 업로드/다운로드가 정상 동작한다.
- [ ] 활동 로그가 사례 등록/수정/삭제 시 정상 기록된다.
- [ ] 필터 검색 (지역, 유형, 단계 등)이 정상 동작한다.

### 역할별 접근 테스트
- [ ] ADMIN 계정으로 모든 기능이 정상 접근된다.
- [ ] MANAGER 계정으로 ADMIN 전용 메뉴가 숨겨진다.
- [ ] MANAGER 계정에서 보상금액·품의 링크가 마스킹된다.
- [ ] 미승인 계정으로 로그인 시 데이터 화면 접근이 차단된다.

### 데이터 정합성
- [ ] 사례 등록 후 DB에 데이터가 정상 저장된다.
- [ ] 사례 삭제 후 `deleted_yn = 'Y'` 로 논리 삭제 처리된다.
- [ ] 공통 감사 컬럼(`created_at`, `updated_at`, `created_by`, `updated_by`)이 정상 기록된다.

### ⚠️ 개발자 검토 필요
- [ ] 보안 항목 전체가 개발자 확인을 받았다 (3단계 보안 체크리스트 참조).
- [ ] DB 스키마가 최종 검토되었다 (1단계 DB 체크리스트 참조).
- [ ] **개발자 최종 검토 완료** ☐

---

## 체크리스트 사용 방법

1. 각 단계 작업이 끝나면 이 문서를 열어 해당 단계 체크리스트를 확인한다.
2. 모든 항목에 `[x]` 체크를 완료해야 다음 단계로 진행한다.
3. **`⚠️ 개발자 검토 완료` 항목**은 개발자에게 확인을 요청하고, 승인 후 체크한다.
4. 미통과 항목이 있으면 해당 항목을 먼저 수정 후 재검증한다.
